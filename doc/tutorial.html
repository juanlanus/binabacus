How I built a jQuery plugin using jquery-boilerplate

jquery-boilerplate

I developed a didactic binary abacus as HTML + CSS + JS inside an HTML page.
It was intended to be featured in my binary numbers micro course.

Once it worked I turned it into a jQuery plugin in order to learn how to build jQuery plugins, a knowledge bit that I lacked and I thought it was gonig to be valuable to me.

After googling for a while I choose to build it using the jQuery Boilerplate scaffolding, mainly because it's the usual JS starting file but with the plus of a complete project structure and wise guidance coming from the masters.
It was (and still is) authored by the great Zeno Rocha with the collaboration of Addy Osmani who is known to be an active JS evangelist and addict.

step by step, instructions and insight

Making a plugin out of my prototype page meant to encapsulate its HTML, CSS and JS so that it would be easy and safe for anybody else to put it to work, without the need to even glance at the code.
Actually, this is the whole idea of a plugin, isn't it?

If you need to know about the inner pinnings of a plugin to use it, then it's not a plugin but sort of a primitive cut & paste code reuse.

I apologize if you find that I'm explaining the obviuos; these are things that once I didn't know and it might be somebody else still not knowing. So please if you find yourself reading about something you already know just skip it, or check it and tell if there are flaws.

Likewise, if there are things you think that should be included please also tell. I'll be grateful.

  12. iife

    The plugin module structure proposed is of the IIFE kind, an Immediately-invoked function expression.
    Those of us who are not familiar with this js module packaging method should do something about it, ASAP.

    The whole module is an anonymous function expression or literal, starting at the first opening parenthesis of the first line, and ending with the first closing parenthesis of the last line.

    The parenthesized expression ( jQuery, window, document ) is the arguments set passed for execution of the anonymous function expression.

    The execution of the IIFE returns an object whose members implement the plugin's functionality.

  13. bootstrapping

    The default binabacus plugin is instantiated thus, on the jQuery ready event: 

              // create a binabacus widget
          $("#binAbacus").binabacus();

    The outcome of these code lines is that a Plugin object is created and a reference to it is stored in the $.data of the container element.

    The binabacus plugin js, closely aligned with the boilerplate sample, does the following:

    • defines a few obvious variables,
    • defines the Plugin function that acts as the constructor,
    • defines the lengthy Plugin.prototype object, with function members to set the audio and the abacus DOM elements, and to furnish then with the proper event handlers, and finally
    • on jQuery's ready event runs the $.fs.binabacus function that starts it all.

    First thing, the function checks that the plugin isn't already instantiated.
    To do so it queries the data property of the plugin container with key jquery_binabacus.
    If the key exists, the function exits silently.

    Otherwise it calls data again, this time with $.fn.the plugin constructor as the third argument.

  15. more HTML

    The abacus also uses an HTML5 <audio> element that, same as the abacus table, is created by the plugin js like so: 

        <!-- the click sound of a ball ending its trip -->
    <audio id="ba_clickSound" preload="auto" volume="0.3" style="display:none">
      <source type="audio/mp3" src="data:audio/ogg;base64,T2dnUwACAAAAAAAAA...
      <source type="audio/ogg" src="data:audio/mp3;base64,//PEBAAAAAAAAAAAA...
    </audio>

    Each source element ends with a lengthy string containing an audio file that produces a "click" sound. Both sources produce the same sound, the difference is the file format. This is so because not all browsers can manage all the audio formats.

  16. data URIs

    As I want the plugin to be simple to implement, I built data URIs for both sound files and stashed them into the plugin as lengthy string variables (audioMP3 and audioOGG), instead of providing separate files.

    This way the plugin is more self-contained and the user doesn't have to remember to copy the sound files along with the js and the css and has less opportunities for frustration.

    The ball background image is also contained into the plugin files, as a data URI in the CSS.

    For the (unlikely) case that a user wanted to use a different sound, the options allow them to specify their own files.

    I built the data URIs using one of the many services available online. Then I broke the sound files in many js lines, to make them safer to handle. It was not possible to do so in the CSS file so the image is a very long code line.

    It turned out, later, that it is possible to inject rules in the live css rules, so this task is pending: set the ball image at initialization time from within the plugin js, if the user hasn't previously set a different one.

  17. pluginize the js

    In the original code some of my jQuery selectors were ids. But I think that a plugin doesn't have the right to polute the ids namespace at will, and that if two instances of the plugin were active at once, it wouldn't work because of the uniqueness requirement for the ids.

    So I replaced the ids with classes and restricted the context of the jQuery search to the current target element (the container), using the second argument of the jQuery/$ function, named "context".

    This way it won't be necessary to advert the user not to use certain ids in her HTML code.

  18. reaching the data from within event handlers

    My widget has event handlers, and it turned out that the settings object was not in the scope of the event handler functions.

    The the initialization function stashed all the information together into the widget container element, thus: 

    $.data( this, 'plugin_' + pluginName, new Plugin( this, settings ) );

    It is accessible with code like this: 

    $( "thecontainer" ).data( "plugin_binabacus" ).settings.someOption )

    This is how the jQuery plugins persist their state between calls.

    The data thing is binded to the container, and my events are binded to elements that are descendants of the container so I have to either blaze a path upwards from the clicked event several levels up to the container, or bind all the events to the container and filter the actions in code. I choose the first option, using jQuery's closest() method.

    I ended up with this: 

      // get a reference to the settings object
    var theSettings = $(this).closest( '.ba_abacus' ).data('plugin_binabacus').settings;

    Placed into an event handler, this line provides access to the settings object because ba_abacus is a class applied to the widget container. This would not work if there were two instances of the plugin in the same page.

  20. now come the bugs
    At this point I can say the plugin is complete but not finished. Committing and pushing.

    Now it's bug time ... :-)

    Actually, there is a bug: albeit the options allow the user to specify an animation time different from the default, it doesn't work.

    It looked simple at first glance: get the value and apply it to the proper DOM elements.

    But it turned out that the value is specified into CSS rules for classes .ba_0 and .ba_1 that are toggled each time a one becomes a zero and viceversa.

    The clean solution is to change the CSS rules programmatically, something I didn't noticed before.

    After googling a litle, I found a page with information and a set of functions to add and delete rules, both for web browsers and the IE.
    It's a short tutorial titled Changing external style sheets using the DOM.

    But I don't need to touch the external styles. Remembering that the "C" in "CSS" stands for cascading, it turns out that I only need to provide a rule at the <style> element level.

    This StackOverflow question provides an usable answer (the one by shadybones)
